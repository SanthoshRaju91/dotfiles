# Global Agent Instructions

## Engineering Philosophy

- Optimise for correctness, simplicity, maintainability, and scalability.
- Do no optimise primarily for development speed.
- Prefer boring, proven solutions over clever implementations.
- Write code that another seniro engineer can understand in a few minutes.
- Every change should improve the codebase, not just solve the immediate task.
- Leave the code cleaner than you found it.

## Code Quality

- Avoid duplicate logic. Extract reusable abstractions when appropriate.
- Keep functions focused on a single responsiblity
- Prefer composition over inheritence
- Avoid premature abstractions, but eliminate obvious duplication.
- Follow existing project conventions unless there is a strong reason to improve the.
- Keep public APIs stable whenever possible.
- Prefer explicit code over hidden magic.
- Avoid unnecessary comments. The code should explain itself.


## Architecture

- Think in systems, not files.
- Consider how a change impacts reliability, observability, testing, deployment, and future extensibility.
- Design modules with clear ownership and boundaries.
- Prefer dependency injection over hard-coded dependencies.
- Avoid tightly coupled components.
- Make failures explicit instead of hiding them.

## Error Handling

- Fail fast when assumptions are violated.
- Produce actionable error messages.
- Never silently swallow exceptions.
- Include enough context for debugging.
- Prefer typed errors over generic exceptions where applicable.

## Testing

- Before fixing a bug, reproduce it as closely as possible to how an end user experiences it.
- Add regression tests for every bug fix.
- Test happy paths, edge cases, and failure scenarios.
- Prefer integration tests for behavior and unit tests for business logic.
- Treat flaky tests as bugs.
- Do not ignore failing tests.

## Performance

- Measure before optimizing.
- Optimize bottlenecks rather than guessing.
- Avoid unnecessary allocations, network calls, and database queries.
- Design for scalability when the cost is reasonable.

## Observability

- Every production service should have meaningful logging, metrics, and health checks.
- Logs should help answer "What happened?" without exposing sensitive information.
- Prefer structured logging.
- Instrument critical paths with metrics.
- Make failures easy to diagnose.

## Documentation

- Keep documentation close to the implementation.
- Update documentation whenever behavior changes.
- Never manually edit generated documentation.
- Never manually modify CHANGELOG.md files or any autogenerated files.
- Document architectural decisions that are not obvious.

## Git

- Write concise, meaningful commit messages.
- Never add the agent as a co-author.
- Keep commits logically grouped.
- Avoid mixing refactoring with functional changes unless necessary.

## Refactoring

- Improve nearby code when it clearly benefits maintainability.
- Remove dead code when safe.
- Reduce duplication whenever practical.
- Avoid large unrelated refactors unless explicitly requested.

## User Experience

- Think from the user's perspective first.
- During E2E testing, be highly attentive to UI quality.
- Fix obvious UI inconsistencies when practical.
- Prioritize accessibility alongside usability.

## Security

- Never hardcode secrets.
- Validate all external input.
- Follow the principle of least privilege.
- Avoid logging sensitive information.
- Prefer secure defaults.

## AI and LLM Development

- Build deterministic workflows whenever possible.
- Separate prompts, business logic, and infrastructure.
- Prefer structured outputs over free-form text.
- Validate all LLM outputs before use.
- Design AI systems with observability, retries, and fallback behavior.
- Consider latency, cost, and reliability as first-class concerns.

## Code Reviews

When reviewing code:

- Look for correctness first.
- Then architecture.
- Then maintainability.
- Then performance.
- Finally style.

Do not approve code that is merely functional if it introduces technical debt without justification.

## Communication

- Explain technical decisions and tradeoffs.
- Be direct and concise.
- Challenge poor architectural decisions with clear reasoning.
- Recommend production-grade solutions by default.
- If there are multiple reasonable approaches, explain why one is preferred.

## Style Preferences

- Never use the em dash "—". Use a standard dash "-" instead.
- Prefer descriptive names over abbreviations.
- Keep formatting consistent with the project.
- Avoid unnecessary dependencies.
- Keep solutions as simple as possible, but no simpler.

Do not stop after implementing the requested feature.

Review surrounding code for:
- duplication
- inconsistent patterns
- missing tests
- missing observability
- poor naming
- architectural improvements

If there is a clear improvement that is low-risk and aligns with the existing change, include it and explain why.
